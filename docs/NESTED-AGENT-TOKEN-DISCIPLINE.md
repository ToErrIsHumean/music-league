1. Nest late: do local inspection, contract checking, and environment preflight before any live nested agent run. Use nested Planner/Implementer/Reviewer calls only for the final evidence pass.
2. Prefer one live pass: default target is one Planner run, one Implementer run, and one Reviewer run. Do not rerun a nested agent unless a concrete blocker or artifact defect requires it.
3. Do not stream by default: start the nested run, wait for completion, and read last-diff.md, review verdicts, and exit status as the primary evidence source. Do not poll for intermediate output or re-request information that already exists on disk.
4. Preflight the harness: before launching a nested agent, verify network access, sandbox mode, artifact paths, git write requirements, and dependency/tool availability in the target workspace. If any of these fail during a nested run, treat it as a preflight defect — resolve it before re-dispatching.
5. Scope review artifacts up front: if the workspace contains unrelated changes, generate a task-scoped diff before invoking the Reviewer. Do not rely on post-hoc diff cleanup.
6. Avoid isolation overhead unless justified: do not create temp worktrees or duplicate environments unless there is a real contamination risk that cannot be managed in the main worktree.
7. Reuse the controlling contract: extract the task entry and its referenced contracts, preserves, and validates once in the outer run. Pass only that narrow surface into nested execution when possible. (This extends TOKEN-DISCIPLINE.md rule 2 to the nested dispatch boundary.)
8. Match validation depth to task size: tiny implementation surfaces should get the minimum nested validation needed to satisfy the contract; do not apply full multi-cycle orchestration by default.
